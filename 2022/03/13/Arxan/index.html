Arxan

tags: Arxan

目的及特色

用來保護app或網頁的資安。產品又分ios及Android

  • 阻止動態攻擊

  • Arxan會去操作.dex(編譯好的程式，即混淆的程式碼和保護機制guard)來做動態guard
    Android App Protection 提供 Java 和 Kotlin 撰寫的 APP 靜態防護、動態防護以及混淆，以防止逆向工程(Reverse engineering)和篡改。
    利用一系列的Guard來做防護。當Guard檢測到攻擊時可以觸發自己定義好的反應機制。或是修改程式碼讓攻擊者難以理解

  • 靜態防護:

    • 防護前端
    • 隱蔽商業邏輯
    • 提高分析難度

  • 動態防護:

    • 運行程式時可以偵測是否受到惡意攻擊並抵擋它

  • 反制攻擊:

    • 可客製化作出相應對反映

A4A 與 加殼

  • A4A
    直接把程式碼的DEX檔抽出來進行混淆後再重新進行封裝(不可逆)
  • 加殼
    在程式碼的外層新增一層保護殼，拆掉這個保護殼後還是可以看到原始沒動過的程式碼。(可逆)

保護流程

  1. 從未防護的APK裡取出他的dex檔
  2. 把這個DEX檔與 A4A 的 GuardSpec 和 Arxan library 一起進行 compile
  3. 得到受到保護的dex檔

原始程式碼+Arxan code(開發完後加)=收到完整防護的app
==我們的任務就是去設計GuardSpec(JSON檔)來保護apk==

A4A Protection重要檔案

:open_file_folder: gs_dir:放防護描述的檔案

  • guardspec.json(防護文件)
  • proguard-rules.properties (Proguard文件)

:open_file_folder:ori_dir: 未防護的APK檔案
:open_file_folder:out_dir:進行防護後產出的相關檔案

  • app-release-unaligned-unsigned-protected.apk ① 剛防護完的 apk (unsigned、unaligned)
  • classes.dex
  • DeobfuscationMap.json
  • ProGuardDeobfuscationMap.txt
  1. 也叫ProGuardMappingFile.txt,在control flow flattening , Inlining,Renaming,Static Member Shuffling時會產生
  2. 在 Guardspce 中的 globalConfiguration 設定可以 input 這個檔
  • protection.log
    A4A 防護時的 CLI 訊息，防護出錯時可以看這裡

:open_file_folder:signed_apk_dir:防護完成並且已經對apk簽章完畢，可以上架的APK

前置作業

==修改 :page_facing_up:runA4A.sh==

  • 設定debug mode和reflection mode

包板時要確定reflection debug 是false

  • 關掉proguard
    在App層的build.gradle裡面將minifyEnable 設為false

關掉proguard的原因是因為，Proguard可以做到的事情A4A 都可以辦到，所以我們只要將所有防護都交給Arxan來做就好了

簽名(Sign)

::: info
允許signing APK 並確認 APK 的簽名在這些 APK 支持的所有 Android 平台版本上成功驗證,由Google提供的Android apk簽名及驗證工具
:::
工具- apksigner
在/Library/Android/sdk/build-tools/<版本號>

v1簽名

  • 來自JDK jarsigner
    v2簽名
  • 來自apksigner
  • 用於Android 7.0
  • 簽名更安全(不能修改压缩)
  • 签名验证时间更短(不需要解压验证),因此安装速度加快

指令

apk 簽名
./apksigner sign --ks “你的keystore”
-ks-key-alias “金鑰別名”
--ks-pass pass:”你的keystore密碼”
--key-pass pass:”你的key密碼”
-out "簽署後的apk位置" 簽署apk後檔案放的位置
